# Railway Deployment Guide — ProxiPro

## Why PostgreSQL instead of SQLite?

SQLite stores its data in a single file on disk. Railway's filesystem is
**ephemeral** — every deployment starts from a clean image, so the SQLite
file is wiped on every deploy and all data is lost. PostgreSQL runs as a
separate, persistent service and is the correct choice for production on
Railway.

SQLite remains the default for **local development** (no extra setup needed).

---

## How the build works

The `railpack.toml` build pipeline handles the SQLite/PostgreSQL split
automatically:

| Phase | What happens |
|-------|-------------|
| **setup** | Copies `.env.example` → `.env` and creates an empty `database/database.sqlite` so Laravel can boot during the build |
| **build** | Runs `composer install`, generates an app key, runs migrations against the build-time SQLite DB, caches routes/views, and builds frontend assets |
| **start** | At runtime Railway injects the real `DB_*` environment variables (PostgreSQL). Migrations run again against the live database before the server starts |

## First-time Railway deployment

### Step 1 — Add a PostgreSQL service

1. Open your Railway project dashboard.
2. Click **+ New** → **Database** → **Add PostgreSQL**.
3. Railway provisions a PostgreSQL instance and exposes connection variables
   (`PGHOST`, `PGPORT`, `PGDATABASE`, `PGUSER`, `PGPASSWORD`).

### Step 2 — Link PostgreSQL to your web service

1. Open your web service → **Variables** tab.
2. Add the following variables (use Railway variable references so they stay
   in sync automatically):

```
DB_CONNECTION=pgsql
DB_HOST=${{Postgres.PGHOST}}
DB_PORT=${{Postgres.PGPORT}}
DB_DATABASE=${{Postgres.PGDATABASE}}
DB_USERNAME=${{Postgres.PGUSER}}
DB_PASSWORD=${{Postgres.PGPASSWORD}}
```

### Step 3 — Set the remaining application variables

```
APP_NAME=ProxiPro
APP_ENV=production
APP_DEBUG=false
APP_URL=https://<your-railway-domain>.up.railway.app
APP_KEY=                        # generated automatically by the build phase
SESSION_DRIVER=cookie
CACHE_STORE=file
QUEUE_CONNECTION=sync
```

> **Note:** `APP_KEY` is generated by `php artisan key:generate --force`
> during the build phase and written to the `.env` file inside the container.
> You do not need to set it manually unless you want a stable key across
> rebuilds (recommended for production — generate one with
> `php artisan key:generate --show` locally and paste it here).

### Step 4 — Deploy

Trigger a new deployment (push a commit or click **Redeploy** in Railway).
The start command runs `php artisan migrate --force` against your PostgreSQL
database before the server starts, so the schema is always up to date.

## Local development

No changes needed. SQLite is still the default for local development:

```bash
cp .env.example .env
# Edit .env and set DB_CONNECTION=sqlite (or leave it as-is if you prefer pgsql locally)
php artisan key:generate
php artisan migrate
php artisan serve
```

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| `SQLSTATE[08006] connection refused` at startup | `DB_*` variables not set or PostgreSQL service not linked | Check Railway variable references in Step 2 |
| `Base table or view not found` | Migrations haven't run | Ensure the start command includes `php artisan migrate --force` |
| Data lost after redeploy | SQLite was used in production | Switch to PostgreSQL following this guide |
| `php artisan package:discover` crash during build | SQLite file missing at build time | The `setup` phase in `railpack.toml` creates it automatically |
